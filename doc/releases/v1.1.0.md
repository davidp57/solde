# Notes de release — v1.1.0

**Date** : 28 avril 2026
**Branche** : `develop` → `main`
**Type** : mineure (nouvelles fonctionnalités, rétrocompatible)

---

## Résumé

Cette release apporte trois axes d'amélioration :

1. **Gestion des dépôts bancaires** — confirmation explicite des bordereaux, état intermédiaire « en transit » pour les chèques, refonte du modèle de dépôt espèces
2. **Améliorations UX** (7 points) — accès paramètres gestionnaires, dialogue paiement enrichi, champs famille sur les contacts, date pré-remplie, commentaires internes, instructions de règlement PDF, protection d'édition facture
3. **Migrations de base de données** : Alembic 0039, 0040, 0041, 0042, 0043

Toutes les modifications sont rétrocompatibles. La mise à jour nécessite un rebuild Docker (migrations auto au démarrage).

---

## Nouvelles fonctionnalités

### Confirmation de dépôt bancaire (BIZ-130)

- Champ `confirmed` (booléen) et `confirmed_date` sur les bordereaux de dépôt (migration Alembic 0038 déjà dans 1.0)
- Endpoint `POST /api/bank/deposits/{id}/confirm` ; audit log `bank.deposit.confirm`
- Panneau « Dépôts en attente de confirmation » dans la vue Banque — liste les bordereaux préparés mais non remis à l'agence, avec résumé (nb chèques ou encaissements, montant) et bouton « Confirmer »
- Colonne « Statut » (en attente / confirmé) dans le tableau des dépôts, avec filtre

### Refonte du modèle de dépôt espèces (BIZ-131)

- Les paiements espèces sont marqués `deposited=True` dès leur création (ils entrent immédiatement en caisse)
- Un bordereau espèces est créé à partir d'un montant et d'un détail optionnel de coupures (billets), sans lien vers des paiements individuels
- La caisse sort (CashEntry OUT) et les écritures comptables sont générées à la **confirmation** du bordereau
- Migration Alembic 0039 : colonne `denomination_details` (TEXT nullable) sur `deposits` ; mise à jour des paiements espèces existants à `deposited=True`
- Métrique « Espèces à déposer » supprimée de la vue Paiements (toujours 0 désormais)

### État intermédiaire « en bordereau » pour les chèques (BIZ-132)

- Un chèque intégré dans un bordereau non confirmé passe à `in_deposit=True, deposited=False` ; `deposited=True` n'est positionné qu'à la **confirmation** (migration Alembic 0040)
- Bouton « Tout sélectionner / Tout désélectionner » dans le dialogue de création de bordereau
- Colonne « Remis en banque » dans la vue Paiements — affiche trois états : remis (✓), en bordereau (horloge orange), à remettre (✗)
- Filtre `undeposited_only` étendu : exclut les chèques déjà en bordereau (état « en transit »)
- Dashboard : le compteur de paiements non remis exclut les chèques en bordereau non confirmé
- Correctif : à la confirmation d'un bordereau espèces, la `BankTransaction` de crédit (entrée banque) est désormais bien créée

### Améliorations UX — Lot H

- **Paramètres en lecture** : les endpoints `GET /settings/` et `GET /settings/system-opening` sont désormais accessibles aux rôles gestionnaire (trésorier, comptable) ; les `PUT` restent réservés aux administrateurs
- **Dialogue paiement enrichi** : affiche le nom du client, la description, le montant total et la date d'échéance de la facture concernée
- **Champs famille sur les contacts** : prénom/nom de l'enfant, prénom/nom de l'autre parent ; la recherche dans le sélecteur de factures inclut ces champs (migrations Alembic 0041)
- **Date du jour pré-remplie** à l'ouverture du formulaire de création de facture client
- **Commentaires internes** (`/comments`) : chaque utilisateur peut saisir des notes/remarques visibles de lui seul ; les admins voient tous les commentaires, peuvent les marquer comme traités et les supprimer (migrations Alembic 0042, 0043)
- **Instructions de règlement dans le PDF** : IBAN, BIC et numéro de chèque de l'association ajoutés en pied de facture, avec le numéro de facture réel dans la référence virement
- **Protection d'édition facture** : l'édition est bloquée si la facture est à l'état `SENT` avec un montant déjà payé, ou à l'état `PAID`
- **PDF épuré** : le bloc Émetteur en doublon du tableau facture a été supprimé

## Migrations de base de données

| Migration | Contenu |
| --- | --- |
| 0039 | Colonne `denomination_details` (TEXT) sur `deposits` ; paiements espèces existants mis à `deposited=True` |
| 0040 | Colonne `in_deposit` (BOOLEAN, défaut `False`) sur `payments` |
| 0041 | Colonnes `child_first_name`, `child_last_name`, `other_parent_first_name`, `other_parent_last_name` (TEXT, nullable) sur `contacts` |
| 0042 | Table `app_comments` : `id`, `user_id` (FK), `content`, `created_at` |
| 0043 | Colonne `is_resolved` (BOOLEAN, défaut `False`) sur `app_comments` |

Les migrations sont appliquées automatiquement au démarrage du conteneur Docker.
